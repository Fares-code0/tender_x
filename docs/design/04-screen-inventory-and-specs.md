## الغرض والمصدر

هذا المستند يوثّق الشاشات الفعلية في تطبيق الويب لنظام إدارة المناقصات (`apps/web/src/pages/*.tsx`)، ويربط كل شاشة بالأدوار التي تصل إليها (`02-roles-permissions-matrix.md`)، والرحلات التي تمر بها (`03-user-journeys.md`، `JRN-xxx`)، وحقولها الفعلية كما تُرسَم في الكود. المعرّفات `SCR-01`...`SCR-07` تُعرَّف رسميًا هنا لأول مرة، وهي المعرّفات نفسها المُشار إليها مسبقًا كمرجع في `03-user-journeys.md`.

> **تحديث M3 + M4:** أصبحت SCR-04 (تفاصيل المناقصة) تحوي **تبويبين** ("التفاصيل" و"المراجعة") و**شريط إجراءات (ActionsBar)** ديناميكيًا أعلى الصفحة يعرض أزرار انتقالات سير العمل حسب (الحالة + الدور). تبويب "المراجعة" (`ReviewTab.tsx`) يعرض الـChecklist التفاعلية وأزرار الاعتماد/الاستبعاد. وأُضيفت شاشة **SCR-07 "مهامي"** (`MyTasksPage.tsx`).

رموز الحالة المستخدمة في جدول الجرد وفي §"الحقول" أدناه:

- ✅ **منفّذ**: الشاشة/الحقل موجود فعليًا في الكود الحالي.
- 🔷 **مخطّط**: عنصر موثّق كمتطلب (غالبًا مرتبط بإجراء 🔷 في `02-roles-permissions-matrix.md` أو بنموذج بيانات موجود في `schema.prisma`) لكنه غير معروض في واجهة الشاشة حاليًا. يُستخدم الرمز نفسه أيضًا داخل جداول الحالات للإشارة إلى حالة شاشة غير مُعالَجة صراحةً في الكود الحالي ومتروكة للتنفيذ لاحقًا.

## SCR-01 — تسجيل الدخول

**الملف:** `apps/web/src/pages/LoginPage.tsx`

**الهدف:** مصادقة المستخدم (بريد إلكتروني + كلمة مرور) والدخول إلى النظام.

**من يراها (الأدوار):** الكل — الشاشة غير محمية بدور؛ إن كان المستخدم مصادَقًا فعليًا بالفعل يُعاد توجيهه تلقائيًا إلى `/` (`<Navigate to="/" replace />`) بدل عرض النموذج.

**الحقول (والإلزامي منها):**

| الحقل | إلزامي؟ | ملاحظات |
|---|---|---|
| البريد الإلكتروني (`email`) | نعم | `type="email"`, `dir="ltr"`, يُتحقق منه عبر `loginSchema` (Zod) قبل الإرسال |
| كلمة المرور (`password`) | نعم | `type="password"`, `dir="ltr"` |

**الرحلات المرتبطة (JRN-xx):** JRN-07 (خطوة "تسجيل الدخول" موثّقة صراحة في جدول خطواتها). تُستخدم ضمنيًا كنقطة دخول قبل أي رحلة أخرى (JRN-01 إلى JRN-06) لكل الأدوار، وإن لم تُدرَج كخطوة صريحة منفصلة في جداول تلك الرحلات.

**جدول الحالات:**

| الحالة | السلوك |
|---|---|
| فارغ | النموذج يُفتح بحقلين فارغين؛ لا تحقق يظهر قبل أول محاولة إرسال (`noValidate` + تحقق Zod عند `onSubmit`) |
| تحميل | أثناء إرسال بيانات الدخول: نص الزر يتحوّل إلى "جارٍ الدخول..." والزر معطَّل (`loginMutation.isPending`) |
| نجاح | إعادة توجيه تلقائي إلى `/` عبر `<Navigate>` بعد نجاح `useLogin` وتحديث حالة `useMe` |
| خطأ | رسالة حمراء أسفل الحقول: خطأ تحقق محلي (رسالة أول Zod issue، أو "بيانات غير صالحة")، أو خطأ خادم (`ApiError.message`)، أو "تعذر الاتصال بالخادم" عند فشل الشبكة |
| عدم صلاحية | غير منطبقة — هذه شاشة الدخول نفسها ولا تتطلب صلاحية دور مسبقة |
| انتهاء الجلسة | غير منطبقة مباشرة — لا توجد جلسة قائمة بعد؛ مستخدم مصادَق فعليًا يُبعَد تلقائيًا عن هذه الشاشة (`Navigate to="/"`) بدل رؤيتها |

## SCR-02 — الرئيسية / لوحة المعلومات

**الملف:** `apps/web/src/pages/HomePage.tsx`

**الهدف:** الترحيب بالمستخدم بعد الدخول وعرض دوره الحالي؛ نقطة انطلاق بعد تسجيل الدخول.

**من يراها (الأدوار):** الكل حسب الدور — أي مستخدم مصادَق (`ADMIN`, `QA`, `WRITER`, `MANAGER`, `OWNER`)؛ لا قيد `roles` إضافي على المسار `/`.

**الحقول (والإلزامي منها):** لا توجد حقول إدخال. محتوى معروض فقط: اسم المستخدم (`user.name`) وتسمية دوره بالعربية (`roleLabels[user.role]`) ونص تعريفي ثابت يفيد بأن "لوحة التحكم الخاصة بدورك ستُبنى في المراحل القادمة (M7)".

**الرحلات المرتبطة (JRN-xx):** JRN-07 (خطوة "فتح الصفحة الرئيسية/لوحة المتابعة" موثّقة صراحة). تُعبَر ضمنيًا بعد الدخول في بقية الرحلات دون أن تُدرَج كخطوة صريحة منفصلة فيها.

**جدول الحالات:**

| الحالة | السلوك |
|---|---|
| فارغ | لا تنطبق فعليًا حاليًا — المحتوى نص ترحيبي ثابت + اسم/دور المستخدم فقط؛ لوحة المعلومات الديناميكية 🔷 مخطّطة لاحقًا (M7) |
| تحميل | لا تُعرض داخل `HomePage` نفسها؛ التحميل الأولي لبيانات المستخدم (`useMe`) يُدار عبر `ProtectedRoute` الذي يعرض "جارٍ التحميل..." قبل الوصول لأي شاشة محمية |
| نجاح | عرض "مرحبًا، {الاسم} ({تسمية الدور})" والنص التعريفي |
| خطأ | لا رسالة خطأ مخصصة داخل `HomePage`؛ الصفحة تُعيد `null` إن لم تتوفر بيانات المستخدم (`if (!user) return null`) — الفشل يُدار فعليًا على مستوى `useMe`/`ProtectedRoute` |
| عدم صلاحية | غير منطبقة — كل الأدوار المصادَقة تصل إليها |
| انتهاء الجلسة | عبر `ProtectedRoute`: فشل `useMe` بخطأ 401 "جلسة غير صالحة، سجّل الدخول مجددًا" يؤدي لإعادة التوجيه إلى `/login` بدل عرض الشاشة |

## SCR-03 — قائمة المناقصات

**الملف:** `apps/web/src/pages/TendersPage.tsx`

**الهدف:** استعراض المناقصات وتصفيتها والبحث فيها، والانتقال إلى تفاصيل مناقصة أو إلى إضافة مناقصة جديدة (لـQA).

**من يراها (الأدوار):** الكل — أي مستخدم مصادَق؛ زر "+ إضافة مناقصة" يظهر فقط إذا كان `user?.role === 'QA'` (عرض شرطي داخل نفس الشاشة، وليس حجبًا للشاشة ذاتها).

**الحقول (والإلزامي منها):** عناصر تصفية وبحث (لا حقول إلزامية — كلها اختيارية وتُطبَّق فورًا):

| العنصر | النوع | ملاحظات |
|---|---|---|
| بحث (`q`) | نص حر | "عنوان أو جهة..."، يُعاد `page` إلى 1 عند التغيير |
| الحالة (`status`) | قائمة منسدلة | خيارات = `TENDER_STATUSES` بتسمياتها العربية (`statusLabels`) + "كل الحالات" |
| الجهة المعلنة (`entity`) | نص حر | مثال: "وزارة الصحة" |
| المسؤول الحالي (`assignee`) | — | 🔷 عمود معروض في الجدول (`currentAssignee`) لكن لا يوجد فلتر واجهي مخصص له حاليًا في الكود |
| موعد الإغلاق (`closingDate`) | — | 🔷 لا يوجد فلتر نطاق تاريخ مخصص حاليًا؛ العمود معروض في الجدول ومُستخدم لحساب "الأيام المتبقية" وتمييز الاقتراب من الإغلاق (`daysUntil`, `activeStatuses`) |
| ترتيب (Sort) | — | 🔷 لا يوجد عنصر تحكم بالترتيب في الواجهة حاليًا؛ الجدول يعرض `closingDate` كعمود لكن دون فرز تفاعلي معروض |
| ترقيم الصفحات (`page`) | ✅ | أزرار "السابق"/"التالي" + عرض "إجمالي النتائج / صفحة X من Y" |

**الرحلات المرتبطة (JRN-xx):** JRN-07 (خطوة "فتح قائمة المناقصات (قراءة فقط)" موثّقة صراحة). تُعد أيضًا نقطة انطلاق فعلية (زر "+ إضافة مناقصة") نحو SCR-05 لبدء JRN-01، وإن لم تُدرَج SCR-03 كخطوة صريحة في جدول JRN-01.

**جدول الحالات:**

| الحالة | السلوك |
|---|---|
| فارغ | "لا توجد مناقصات مطابقة" ضمن إطار منقّط عندما `data.tenders.length === 0` |
| تحميل | "جارٍ التحميل..." (`isLoading`) |
| نجاح | عرض جدول المناقصات (العنوان، الجهة، الحالة، موعد الإغلاق، المسؤول الحالي) + الترقيم |
| خطأ | "تعذر تحميل المناقصات" (`error != null`) |
| عدم صلاحية | لا حجب على مستوى الشاشة نفسها؛ زر "+ إضافة مناقصة" مخفي لغير QA (تحكم واجهي فقط)؛ محاولة الوصول المباشر لـ`/tenders/new` من دور غير QA تُرفض عبر `ProtectedRoute` (انظر SCR-05) |
| انتهاء الجلسة | فشل الاستعلام (`fetchTenders`) بخطأ 401 "جلسة غير صالحة، سجّل الدخول مجددًا" يؤدي لإعادة التوجيه إلى `/login` عبر `ProtectedRoute`/`useMe` |

## SCR-05 — نموذج المناقصة (إنشاء/تعديل)

**الملف:** `apps/web/src/pages/TenderFormPage.tsx`

**الهدف:** إنشاء مناقصة جديدة (QA) أو تعديل بيانات مناقصة قائمة (QA/MANAGER/ADMIN)، بتحقق Zod مطابق للـbackend ورسائل عربية.

**من يراها (الأدوار):** الإنشاء (`mode="create"`, المسار `/tenders/new`) — QA فقط. التعديل (`mode="edit"`, المسار `/tenders/:id/edit`) — QA / MANAGER / ADMIN.

**الحقول (والإلزامي منها):**

| الحقل | إلزامي؟ | ملاحظات |
|---|---|---|
| عنوان المناقصة (`title`) | نعم | علامة `*` حمراء في التسمية |
| الجهة المعلنة (`entity`) | نعم | علامة `*` حمراء؛ BR-010 |
| موعد الإغلاق (`closingDate`) | **نعم — BR-010** | `type="date"`، علامة `*` حمراء |
| المصدر (`source`) | لا | مثال: "منصة اعتماد" |
| رابط المناقصة (`url`) | لا | `dir="ltr"` |
| الوصف (`description`) | لا | `textarea` متعدد الأسطر |

**الرحلات المرتبطة (JRN-xx):** JRN-01 (تسجيل مناقصة) — كل خطواتها الثلاث الأولى (فتح النموذج، إدخال البيانات الإلزامية، تحذير التكرار وتأكيد التجاوز) تمر عبر هذه الشاشة.

**جدول الحالات:**

| الحالة | السلوك |
|---|---|
| فارغ | نموذج فارغ (`emptyForm`) عند الإنشاء؛ في وضع التعديل تُملأ الحقول تلقائيًا من بيانات المناقصة القائمة (`existing.data`) |
| تحميل | وضع التعديل: تحميل بيانات المناقصة الحالية (`existing` query)؛ عند الإرسال: نص الزر "جارٍ الحفظ..." والزر معطَّل (`mutation.isPending`) |
| نجاح | إعادة توجيه إلى `/tenders/:id` (صفحة التفاصيل) بعد `onSuccess` + إبطال كاش `tenders`/`tender` |
| خطأ | رسائل تحقق حقلية تحت كل حقل (`fieldErrors` من Zod)؛ رسالة خادم عامة حمراء (`serverError`)؛ أو تحذير تكرار برتقالي (استجابة 409 → `duplicateWarning`) مع زر "تأكيد الإضافة رغم التكرار" (`force`) |
| عدم صلاحية | الوصول لـ`/tenders/new` مقصور على QA فقط، ولـ`/tenders/:id/edit` على QA/MANAGER/ADMIN — أي دور آخر يُعاد توجيهه بواسطة `ProtectedRoute` قبل رؤية النموذج؛ وعلى مستوى الخادم: "ليست لديك صلاحية لهذا الإجراء" (403) |
| انتهاء الجلسة | إرسال النموذج بجلسة منتهية يُعيد 401 "جلسة غير صالحة، سجّل الدخول مجددًا" (يظهر ضمن `serverError`) |

## SCR-06 — إدارة المستخدمين

**الملف:** `apps/web/src/pages/AdminUsersPage.tsx`

**الهدف:** عرض قائمة مستخدمي النظام مع أدوارهم وحالة تفعيلهم (إدارة المستخدمين والأدوار، ACT-12).

**من يراها (الأدوار):** ADMIN فقط — المسار `/admin/users` محمي بـ`ProtectedRoute roles={['ADMIN']}`؛ رابط التنقل إليها في `Layout` لا يظهر أصلًا لغير ADMIN.

**الحقول (والإلزامي منها):** لا توجد حقول إدخال/تعديل في هذه الشاشة حاليًا (عرض للقراءة فقط):

| العمود | ملاحظات |
|---|---|
| الاسم (`name`) | عرض فقط |
| البريد الإلكتروني (`email`) | عرض فقط، `dir="ltr"` |
| الدور (`role`) | تسمية عربية عبر `roleLabels` |
| الحالة (`isActive`) | شارة "نشط" (أخضر) أو "معطّل" (أحمر) |

**الرحلات المرتبطة (JRN-xx):** لا توجد رحلة موثّقة في `03-user-journeys.md` لهذه الشاشة — الإجراء `ACT-12` (إدارة المستخدمين والأدوار) لا يقابله `JRN` مخصص ضمن نطاق حزمة رحلات المستخدمين الحالية (التي تغطي دورة حياة المناقصة تحديدًا).

**جدول الحالات:**

| الحالة | السلوك |
|---|---|
| فارغ | 🔷 غير مُعالجة صراحة في الكود الحالي — لا توجد رسالة "لا يوجد مستخدمون" مخصصة؛ إن كانت `data.users` فارغة يُعرض جدول بلا صفوف |
| تحميل | "جارٍ التحميل..." (`isLoading`) |
| نجاح | عرض جدول المستخدمين (الاسم، البريد، الدور، الحالة) |
| خطأ | "تعذر تحميل المستخدمين" (`error`) |
| عدم صلاحية | الوصول مقصور على ADMIN فقط عبر `ProtectedRoute`؛ أي دور آخر يُعاد توجيهه إلى `/` قبل الوصول للشاشة؛ وعلى مستوى الخادم مسارات `/admin/users` تعيد "ليست لديك صلاحية لهذا الإجراء" (403) لغير ADMIN |
| انتهاء الجلسة | فشل تحميل المستخدمين بخطأ 401 "جلسة غير صالحة، سجّل الدخول مجددًا" وإعادة التوجيه لتسجيل الدخول |

## SCR-07 — مهامي (My Tasks)

**الملف:** `apps/web/src/pages/MyTasksPage.tsx`

**الهدف:** عرض المناقصات المعنيّة بالمستخدم الحالي حسب دوره، كنقطة انطلاق سريعة لمهامه في سير العمل.

**من يراها (الأدوار):** QA / WRITER / MANAGER — رابط "مهامي" في `Layout` يظهر لهذه الأدوار الثلاثة فقط (لا يظهر لـOWNER/ADMIN). المسار `/my-tasks` محمي بـ`ProtectedRoute` (أي مستخدم مصادَق).

**منطق العرض حسب الدور:**

| الدور | ما يُعرَض | الاستعلام |
|---|---|---|
| QA / WRITER | المناقصات المعيّنة له (`currentAssignee = أنا`) | `GET /tenders?assigneeId={me}` |
| MANAGER | المناقصات بانتظار الاعتماد (مجمّع المدراء) | `GET /tenders?status=PENDING_APPROVAL` |

**الأعمدة:** العنوان (رابط لـSCR-04)، الجهة المعلنة، الحالة (`StatusBadge`)، موعد الإغلاق (تمييز أحمر عند اقتراب الإغلاق للحالات النشطة).

**الرحلات المرتبطة (JRN-xx):** يخدم بشكل عرضي JRN-03 (الكاتب يجد المناقصة المعيّنة له) وJRN-04/JRN-05 (المدير يجد ما بانتظار اعتماده)؛ ونُفِّذ التحقق منه في بوابة خروج M4.7.

**جدول الحالات:**

| الحالة | السلوك |
